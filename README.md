# API Guardian

Detects breaking changes between two versions of an OpenAPI spec, scores the
overall risk, and (once you add the AI layer) explains changes in plain
English and generates migration guides / release notes.

## What's built (verified working end-to-end)

- **Diff engine** (`app/diff/engine.py`) — deterministic, no AI involved.
  Detects: added/removed endpoints, added/removed fields, type changes,
  required↔optional changes, enum narrowing/widening, description changes.
  Each change gets a rule-based severity: safe / medium / high / critical.
- **Parser** (`app/parser/openapi_parser.py`) — loads YAML or JSON into a
  plain dict for the diff engine.
- **Models** (`app/models/models.py`) — User, Project, SpecVersion,
  Comparison, ChangeRow, AIReport. Note: `ChangeRow.ai_explanation` and
  `AIReport` are nullable/separate from the diff engine on purpose — the
  diff is always correct independent of whether the AI layer has run yet.
- **API routes** (`app/api/routes.py`) — create project, upload spec
  version, run comparison, fetch comparison result.

Tested: create project → upload two spec versions → compare → get back a
structured JSON report with per-field changes and an overall risk score.
Ran through the real FastAPI HTTP layer against a SQLite database, matching
the "name/email removed" example from the spec.

## What's NOT built yet (your next steps)

1. **Auth** — `get_current_user_id()` in `routes.py` is a hardcoded stub.
   Wire up JWT + Google OAuth before this touches real users.
2. **AI layer** (`app/ai/` — currently empty) — three features to build:
   - `explain_change(change: ChangeRow) -> str` — plain-English explanation
   - `generate_migration_guide(comparison: Comparison) -> str`
   - `generate_release_notes(comparison: Comparison) -> str`
   These should run as background jobs (Celery + Redis), not inline in the
   request — LLM calls are slow and you don't want to block the comparison
   response on them. The comparison result should return immediately with
   `ai_explanation: null`, then the frontend polls or gets a websocket push
   once it's ready.
3. **Frontend** — nothing built yet. Next.js + Tailwind + shadcn/ui as
   planned. Key screens: project list, spec upload, diff report (GitHub-style
   +/- view), risk dashboard.
4. **GitHub Actions style CLI** — a `npx`-style CLI wrapping the compare
   endpoint would make for a great demo gif in your README.

## Running it locally

```bash
cd backend
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# for local dev without a real Postgres instance:
export DATABASE_URL="sqlite:///./dev.db"
# for real use:
# export DATABASE_URL="postgresql://user:pass@localhost:5432/api_guardian"

uvicorn app.main:app --reload
```

Then visit `http://localhost:8000/docs` for interactive API docs (auto-
generated by FastAPI from the routes — a nice bit of dogfooding, since this
whole product analyzes OpenAPI specs).

## Design notes worth remembering for interviews

- **Severity is a lookup table** (`SEVERITY_RULES` dict in `engine.py`), not
  scattered if/else logic. Easy to unit test, easy to explain, easy to tune.
- **AI never decides what's breaking** — it only explains what the
  deterministic engine already decided. If asked "what happens if the AI
  hallucinates," the answer is: the diff and severity are unaffected.
- **UUIDs are stored as `String(36)`**, not the Postgres-specific UUID type —
  keeps SQLite (tests, local dev) and Postgres (production) behaving
  identically, which is what let this get tested quickly without spinning up
  a real database.
